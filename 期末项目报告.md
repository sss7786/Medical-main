## 摘要

本项目在完成课程「智能体 + Web 集成」要求的前提下，实现了一个面向门诊前的**信息整理与备忘** Web 应用：用户通过自然语言对话、分步表单与 PDF 上传等方式录入症状与时间线等信息，后台 **FastAPI 智能体** 调用大语言模型（LLM）进行多轮澄清，并将内容整理为结构化 JSON 报告；同时具备 **PDF 文本提取工具**、按会话的 **记忆合并**、危急表述 **安全拦截** 与前端 **流式输出** 等工程能力。系统强调**非诊断、非治疗建议**，贴合医疗场景伦理与课程对 Agent 能力维度的考核。

**关键词：** 智能体；FastAPI；React；大语言模型；工具调用；会话记忆；流式对话

## 1. 项目背景与目标

### 1.1 背景

门诊沟通时间有限，患者常需要在就诊前自行梳理症状、用药与过敏史。纯文本聊天或零散记录不利于携带与复盘。结合课程要求，本项目将「LLM 对话能力」与「结构化输出、Web 交互」结合，构建可演示、可扩展的智能体应用。

### 1.2 项目目标（对齐考核「核心功能」）

| 课程要求 | 本项目对应目标 |
|----------|----------------|
| 智能体具备 LLM 理解与生成 | 多轮问诊式中文/英文对话，就诊前信息澄清 |
| 工具（Tools） | PDF 文本抽取；结构化报告合成（JSON）；可选真实/离线 LLM 网关 |
| 记忆（Memory） | 按 `session_id` 持久化摘要（过敏、用药等），并回注后续对话 |
| 规划与推理（Planning & Reasoning） | 系统 Prompt 分步引导症状、时间线、用药、过敏；表单与对话联动 |
| Web 集成 | React 单页应用为统一入口，与后端 REST/NDJSON 流式接口深度集成 |

### 1.3 应用边界与伦理声明

本系统**仅辅助就诊前信息整理**，不提供诊断、治疗方案或用药指导；当用户输入匹配危急关键词时，后端中止常规模型流程并返回**立即就医/急救**提示。课程演示数据不应包含真实个人隐私。

## 2. 技术选型

### 2.1 总体架构

采用 **前后端分离**：前端 **Vite + React + TypeScript**；后端 **Python 3.10+**、**FastAPI**。未使用 LangChain / CrewAI 等框架，而以 **原生 FastAPI 路由编排** LLM 调用、会话与工具逻辑，依赖清晰，便于答辩时对照代码讲解（符合课程「允许 Flask/FastAPI」等选型说明）。

### 2.2 智能体与模型

- **对话与报告生成：** 兼容 OpenAI 协议兼容网关（如 OpenAI、DeepSeek 等），通过环境变量配置 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL`。
- **无 Key 降级：** 未配置 API Key 时，对话采用内置话术轮换，报告使用离线占位 JSON，保证作业可本地演示。

### 2.3 前端技术要点

- UI 组件与交互（按钮、布局、侧栏报告预览等）
- **i18n**（中/英）与多**接诊场景**（如全科/儿科/慢病）配置
- 对接 **`POST /api/chat/stream`** 的 NDJSON 流式解析，实现打字机效果
- 可选 **Web Speech API** 语音输入（推荐 Chrome / Edge）
- 浏览器 **打印/PDF** 导出（用户通过系统打印对话框另存为 PDF）

### 2.4 后端与数据

- **接口：** `POST /api/chat`、`POST /api/chat/stream`、`POST /api/synthesize`、`POST /api/extract-pdf`，以及会话 `GET/POST /api/sessions` 等
- **记忆存储：** `server/data/memory/` 下按会话 JSON 文件（提交 zip 时请不包含个人隐私数据）
- **依赖说明：** `server/requirements.txt`；前端 `package.json`（评审按文件重装依赖）

### 2.5 与课程「推荐栈」的对照说明

课程推荐 Django + 原生/框架 Agent；本项目选用 **FastAPI + 自研编排**，理由：异步友好、与流式 HTTP 响应契合、自动 OpenAPI 文档（`/docs`）便于验收与答辩。能力维度仍覆盖 **LLM、Tools、Memory、规划与安全规则**。

## 3. 智能体设计（对应评分「智能体设计与创新性」）

### 3.1 角色与系统提示（Planning）

后端通过 **系统 Prompt** 约束智能体角色为「就诊前信息整理助手」：分步澄清**症状、持续时间、用药、过敏史**等，语气克制，禁止诊断与处方建议。危急关键词由正则预先拦截，不进入常规模型链路。

### 3.2 工具（Tools）设计

| 工具能力 | 接口 | 作用 |
|----------|------|------|
| 多轮对话（非流式） | `POST /api/chat` | 完整回复一次返回 |
| 多轮对话（流式） | `POST /api/chat/stream` | NDJSON 分块输出，提升交互体验 |
| PDF 文本提取 | `POST /api/extract-pdf` | 从上传 PDF 抽取文本供用户粘贴或后续整理 |
| 结构化报告合成 | `POST /api/synthesize` | 将对话/表单要点整理为 JSON（含时间线、用药、过敏、就诊科室建议字段等） |

`GET /api/health` 返回的 `agent` 字段中可对工具与记忆能力做机器可读说明，便于验收演示。

### 3.3 记忆（Memory）机制

- 每个 `session_id` 对应一份记忆文件，保存过敏、用药摘要等结构化摘要。
- **合成报告后**将结果中的过敏/用药等 **合并** 回记忆（`_merge_memory_from_bundle`），使后续轮次对话在上下文外仍保有「长期要点」。
- **对话请求**前将记忆摘要以额外 system 消息注入，实现「跨轮次一致的个性化追问」。

### 3.4 流式输出与 Web 体验（对应「WebSocket 举例」）

课程示例提到 WebSocket；本项目采用 **HTTP 长连接 + NDJSON 流**，在工程上易与 CDN/负载均衡兼容，同样满足「流式反馈」的评分要点：前端逐 token/块更新气泡，降低首字延迟。

### 3.5 创新性说明（可答辩强调）

- **多场景接诊模板**（前端）：不同场景下欢迎语、表单快捷项与问诊进度关键词不同，体现任务型对话的产品化设计。
- **伦理与安全**：急诊拦截 + 明确非诊断定位，契合医疗类 Agent 社会责任表述。

## 4. 系统实现细节

### 4.1 目录与模块（代码质量与工程规范）

- `server/main.py`：路由、LLM 调用、记忆、合成、PDF、危急检测
- `server/tests/`：关键逻辑单元测试（如危急检测、消息拼装等）
- `src/app/`：页面与 `ChatInterface`、头部导出、面板布局
- `src/utils/consultScenarios.ts`：场景与问诊进度条关键词逻辑
- `README.md`：安装、运行、构建与测试命令

### 4.2 前后端协作流程（简述）

1. 用户打开 Web，创建或使用已有 `session_id`。
2. 用户输入或语音转写 → 前端调用 `POST /api/chat/stream` → 实时展示助手回复。
3. 用户触发「刷新报告」→ `POST /api/synthesize` → 右侧/侧栏展示结构化 JSON 预览；记忆持久化更新。
4. 可选：上传 PDF → `POST /api/extract-pdf` → 文本回填到对话或备忘。

### 4.3 关键实现说明

- 开发环境下前端通过 **Vite 代理** `/api` 到 `127.0.0.1:8000`，减少 CORS 与 IPv6 混用问题。
- 生产环境通过 `VITE_API_BASE` 指向公网后端。

## 7. 问题与解决（对应评分「实现过程」）

| 问题 | 现象 | 解决思路 |
|------|------|----------|
| 流式对话时 UI 异常 | 父组件频繁刷新 `initialMessages` 导致重复气泡 | 取消不当的 `initialMessages → messages` 全量同步；稳定消息 id |
| 进度条不随对话更新 | 仅按用户文本匹配 | 从首条用户消息起合并用户+助手全文做关键词匹配；流式更新 `messages` |
| 菜单/下拉与扩展冲突 | Radix 下拉在部分环境点击失效 | 场景与语言改用原生 `<select>`；导出菜单用 `Popover` 等更稳妥交互 |
| 模型网关 405 | Base URL 拼接到错误 path | 统一 `build_chat_completions_url` 规则；文档中说明 DeepSeek/OpenAI 差异 |
